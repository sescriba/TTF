\section{Pruebas funcionales del hardware}
\label{sec:pruebasHW}
Esta sección expone las pruebas realizadas sobre el hardware del sistema. Cabe recordar que este proyecto es un prototipo funcional, por lo que las pruebas ejecutadas son para validar funcionamiento básico del sistema.

\subsection{Pruebas funcionales de los sensores PIR}
Para probar el funcionamiento de los sensores PIR conectados al sistema, se procede a alimentar, de acuerdo a las especificaciones técnicas, con tensión y corriente al dispositivo. Por otro lado, se conecta al pin de salida digital un LED de baja potencia junto con una resistencia de 1 K$\Omega$ para verificar las señal generada.

\subsubsection{Análisis de sensibilidad}
Se coloca el sensor PIR de manera estable sobre el borde de una mesa y el usuario procede a realizar un movimiento frente a él, el cual se ejecuta a diferentes distancias del sensor, 30 cm, 1 m y 2 m. De esta manera se pretende verificar que el LED se encienda al movimiento y medir la sensibilidad del dispositivo.
\begin{itemize}
    \item {30 cm}: el LED enciende correctamente. 
    \item {1 m}: el LED enciende con intermitencias. Se ajusta la sensibilidad y enciende correctamente.
    \item {2 m}: el LED enciende con intermitencias. Se ajusta la sensibilidad y enciende correctamente.
\end{itemize}
Cabe destacar que una vez ajustada la sensibilidad, se procede a repetir las pruebas previas a fin de comprobar que el LED aún enciende correctamente frente al cambio de sensibilidad.
\subsubsection{Detección de señal y uso de transistores}
Se conexiona el sensor PIR de acuerdo a la figura \ref{fig:transistor}, se lo coloca de manera estable sobre el borde de una mesa y el usuario procede a realizar un movimiento frente a él. Se verifica que el LED permanece encendido cuando no hay movimiento y se apaga al ejecutarse el movimiento.

\subsection{Pruebas funcionales de los extensores de puertos I\textsuperscript{2}C}
Como ya se mencionó en la subsección \ref{sec:extensorpuertos}, los extensores de puertos I\textsuperscript{2}C envían un byte de información con el estado general de todos sus puertos. Para las pruebas a continuación se procede a realizar un pequeño firmware de test donde el microcontrolador sirve como maestro y el extensor de puertos como esclavo.
\subsubsection{Análisis de trama de datos mediante protocolo I\textsuperscript{2}C}
Se conecta a los pines de SCL y SDA un analizador lógico digital a fin de capturar una trama de requerimiento de información por parte del maestro y la respuesta correspondiente por parte del esclavo. Para este caso, se conecta el pin 1 del extensor a tierra y los demás pines a 5 V. Observando la figura \ref{fig:trama} se puede verificar el correcto funcionamiento del extensor de puertos al obtener una trama respuesta acorde a lo solicitado.

\subsection{Pruebas funcionales de los LEDs}
\label{sec:pruebaled}
A continuación se procede a verificar el correcto funcionamiento de la tira LED utilizada en el sistema. Se realizan pruebas de consumo tanto de tensión como de corriente y se verifica el correcto encendido de los mismos.
\subsubsection{Prueba de encendido}
Para esta prueba se desarrolla un pequeño firmware en el microcontrolador para producir el encendido de los LEDs mediante PWM. Al ejecutar el firmware los LED se encienden produciendo una tonalidad blanca, verificando el correcto funcionamiento de ellos. 
\subsubsection{Prueba de consumo de tensión y corriente}
Utilizando el mismo firmware de la prueba anterior, se procede a encender los LEDs, conectar un multímetro sobre el primer LED y el último del circuito y a medir su tensión. Luego, mediante una pinza amperométrica, se procede a medir la corriente consumida. Las mediciones arrojan 5 V de tensión sobre el primer y último LED, y una corriente de consumo de 3,9 A.

\section{Pruebas funcionales del software}
\label{sec:pruebasSW}
Las pruebas funcionales de software persiguen el objetivo de validar el firmware desarrollado para el sistema. De esta manera, se ejecutan pruebas unitarias mediante el uso del depurador y pruebas generales de secciones de código implementado.

\subsection{Pruebas funcionales sobre la tarea de captación de datos}
Para este conjunto de pruebas se hace uso del hardware finalizado de acuerdo al esquemático de la figura \ref{fig:schematic} y verificado en la sección \ref{sec:pruebasHW}. Al ejecutar las pruebas a continuación, se valida de manera complementaria el correcto funcionamiento de la iteración de la tarea del sistema operativo.
\subsubsection{Pruebas de funcionamiento sobre la comunicación I2C}
Haciendo uso del depurador se verifica que la inicialización del periférico I2C se ejecuta correctamente al ser llamado desde la tarea de control mediante el estado \texttt{SMPowerUp}. Luego de retornar correctamente, el sistema itera llamando a la función de obtención de datos. Sin movimiento frente al sensor PIR, se valida que el dato recibido es el mismo sin variaciones. Al cambiar el estado del sensor PIR mediante un movimiento frente a él, se valida el cambio de información recibida.

También se valida el correcto cambio de la bandera global de sincronización. Al enviar el comando de lectura, el depurador muestra que esta cambió a estado \texttt{ocupado}. Luego, al recibir el dato, a estado \texttt{listo}. Finalmente se verifica que los datos se hayan posprocesados y almacenado en el arreglo para su transferencia a la tarea de control con el formato correspondiente.

\subsubsection{Pruebas de funcionamiento sobre la comunicación UART}
Haciendo uso del depurador se verifica que la inicialización del periférico UART se ejecuta correctamente al ser llamado desde la tarea de control mediante el estado \texttt{SMPowerUp}. Luego de retornar correctamente, el sistema itera llamando a la función de obtención de datos. Al enviar un dato desde la terminal se puede validar la correcta recepción del mismo mediante un buffer de almacenamiento, correspondiendo a lo enviado por el usuario. 

También se valida el correcto cambio de la bandera global de sincronización. Al enviar el comando de lectura, el depurador muestra que esta cambió a estado \texttt{ocupado}. Luego, al recibir el dato, a estado \texttt{listo}. Finalmente se verifica que los datos se hayan posprocesados y almacenado en el arreglo para su transferencia a la tarea de control con el formato correspondiente.

\subsubsection{Pruebas de funcionamiento sobre colas de comunicación entre tareas}
Una vez obtenidos los datos y posprocesados y se los envía mediante la cola, se verifica mediante el uso del depurador, que los mismos fueron recibidos en el formato correspondiente en la tarea de control. También se valida que las banderas globales de sincronización cambian acorde a lo estimado.

\subsection{Pruebas funcionales sobre la tarea de control}
La tarea de control es uno de los principales bloques de código a verificar, ya que es quién comanda las acciones y decisiones del sistema y aloja la máquina de estados.

\subsubsection{Pruebas de funcionamiento sobre la máquina de estados}
Para verificar el correcto funcionamiento de la máquina de estados se procede a hacer uso del depurador y leer la variable global de estado actual de la máquina. Ya validada la tarea de obtención de datos y recepción de estos mediante las colas de sincronización, se analiza el flujo de la máquina de estados y cómo los mismos cambian a medida que la información varia.

En primer término se verifica que el estado por defecto de la variable de estado es \texttt{SMPoweredDown} al iniciar el sistema. Luego se envía un mensaje mediante la UART para dar inicio a la ejecución del flujo y se verifica que el estado cambió a \texttt{SMPoweredUp}. Finalmente se verifica que los LEDs se encuentran apagados. Se repite el caso pero en este caso se fuerza a fallar la función de inicialización, obligando a la máquina de estado a ir a \texttt{SMModulePoweredDown}. Los LEDs no cambian su estado previo, y se valida que el sistema se deshabilitó y los periféricos fueron desinicializiados.

En segundo lugar, luego de que el se verifica que el flujo sigue correctamente, se procede a enviar un mensaje de encendido, cambiando el estado a \texttt{SMRun} y encendiendo los LEDs en color blanco. Se repite la prueba forzando una falla en el encendido de los LEDs y validando que la máquina de estados cambia a \texttt{SMStop} y luego a \texttt{SMModulePoweredDown} y el sistema se deshabilita y los LEDs se apagan.

El siguiente paso consiste en enviar un mensaje de deshabiltiación, forzando a ir al estado de \texttt{SMStop}. Los LEDs se apagan y cambia el estado a \texttt{SMPoweredUp}. Se valida también que el sistema queda habilitado a volver a iterar y lograr ir al estado \texttt{SMRun} repitiendo los pasos anteriores. Se fuerza una falla para que el sistema vaya a \texttt{SMModulePoweredDown} dehabilitandolo.

Para el cambio de color, se enciende el sistema en color blanco y se envía el comando correspondiente para cambiar el color de los LEDs a rojo. En la figura \ref{fig:rojo} se puede apreciar el cambio.

Finalmente se envían los comandos para cambiar el modo de configuración. Se reinicia el sistema y se mantiene la máquina de estados en \texttt{SMRun}. En la lista a continuación se detallan las pruebas.
\begin{itemize}
    \item {Apagado de los LEDs}: se envía comandos para apagar los LEDs, verificando que los mismos se apagan. 
    \item {Encendido de los LEDs y mantiene}: se envía un comando para encender los LEDs a medida que el sensor detecta el movimiento y los mantiene encendidos, verificando el funcionamiento.
    \item {Encendido de los LEDs y no mantiene}: se envía un comando para encender los LEDs a medida que el sensor detecta el movimiento pero no los mantiene encendidos, apagándolos al cabo de un tiempo. Se verifica el funcionamiento.
    \item {Apagado de los LEDs y mantiene}: se envía un comando para apagar los LEDs previamente encendidos a medida que el sensor detecta el movimiento y los mantiene apagados, verificando el funcionamiento.
    \item {Apagado de los LEDs y no mantiene}: se envía un comando para apagar los LEDs a medida que el sensor detecta el movimiento pero no los mantiene apagados, encendiéndolos al cabo de un tiempo. Se verifica el funcionamiento.
\end{itemize} 
Las pruebas anteriores verificaron correctamente el funcionamiento del sistema.

\section{Prueba de funcionamiento general del sistema}
Finalmente, se verifica el funcionamiento general del hardware y software de manera integra. Para este caso se conectan los dispositivos de acuerdo al esquemático de la figura \ref{fig:schematic} mientras se corre en el firmware final desarrollado. Se envían diferentes comandos desde la terminal UART mientras se fuerza un movimiento en los sensores PIR, verificando que los comandos cambian de color, brillo y modo de trabajo correctamente. De manera paralela se verifica que dichos comandos se pueden ejecutar mientras los sensores detectan correctamente los movimientos del usuario.
